# Jira工单统计技能 V2.6

## 功能
统计指定release版本、指定类型的工单修复时间，按组件维度输出平均/最短/最长修复时间，支持可视化柱状图展示。统计效率提升10倍，数据100%准确。**新增：评论情况统计 + 可视化PDF报告生成**。

## 重要：交互式确认流程

当用户请求查询**未关闭BUG**时，必须按以下流程操作：

### 步骤1：询问用户确认筛选条件
首先告知用户默认的筛选条件，并询问是否需要修改：
```
🔍 默认筛选条件：
- status: IN (In-Progress, NEW, In-Verify, Blocked)
- issuetype: BUG_
- priority: Highest

请确认是否正确？如有需要请修改。
```

### 步骤2：等待用户确认
用户可能：
1. **确认正确** → 直接执行查询
2. **修改条件** → 使用用户提供的条件执行查询
3. **不确定** → 提供常见状态选项供选择

### 步骤3：执行查询
确认筛选条件后，再执行JQL查询。

### 常见状态选项（供用户选择）
- 未关闭：In-Progress, NEW, In-Verify, Blocked
- 已关闭：Closed, Done, Resolved, Duplicate, Postponed
- 所有状态：空（不筛选）

## 用法
### 1. 修复时间统计
```
统计 <release名称> <工单类型> [priority级别] 的修复时间
# 示例：
统计 alpha release 1.0.1 BUG_ 的修复时间
统计 alpha release 1.0.1 BUG_ highest 的修复时间
统计 1.0.0版本 Story High 的处理时间
```

### 2. 未关闭问题查询
```
查询 <release名称或多个版本> 未关闭的 <priority级别> 问题
# 示例：
查询 alpha release 1.0.1 未关闭的 Highest 问题
查询 beta release 2.0.0 未关闭的 Highest 问题
查询 alpha release 1.0.1 和 beta release 2.0.0 未关闭的 Highest 问题
查询 gamma release 1.5.0 未关闭的 High 问题
```

**注意**：未关闭的正确筛选条件为：
- status IN (In-Progress, NEW, In-Verify, Blocked)
- priority = Highest（或其他级别）
- issuetype = BUG_
- fixVersion = 用户指定的版本（支持多版本用逗号分隔）

### 3. assignee切换分析
```
统计 <release名称> assignee切换超过3次的工单
# 示例：
统计 alpha release 1.0.1 assignee切换超过3次的工单
```

### 4. 评论情况统计（V2.5新增）
```
统计 <release名称> <priority级别> 评论情况
# 示例：
统计 M1000 release 1.4.0 Highest 的评论情况
统计 alpha release 1.0.1 High 的评论间隔
```

**评论统计输出内容**：
- 首次评论间隔：从ticket创建到第一条评论的天数
- 平均评论间隔：所有相邻评论之间天数的平均值
- 按Component维度汇总
- 每个ticket的详细数据

**重要**：评论统计需要展开每条评论获取精确时间戳，计算量较大。
- 30个ticket约需1-2小时
- 需要每5分钟汇报进度
- 评论数多的ticket（如30+条）需要更长时间

### 5. 生成可视化PDF报告（V2.6新增）
评论统计完成后，自动生成可视化PDF报告：
- Component维度汇总表格
- 柱状图（按从高到低排序，数据标注）
- 30个ticket详细精确数据表格
- 关键发现总结

PDF文件自动命名为：`<release>_<类型>_评论报告_YYYYMMDD_HHMM.pdf`

支持的priority级别：Highest, High, Medium, Low, Lowest

## 输出格式
```

| 组件 | 工单数量 | 平均修复时间(小时) | 最短修复时间(小时) | 最长修复时间(小时) | 最长修复工单 |
|------|----------|-------------------|-------------------|-------------------|--------------|
| 组件名 | 数量 | 平均 | 最短 | 最长 | SW-XXXX: 问题描述 |

## 整体汇总
- 总工单数量: X
- 整体平均修复时间: X 小时（约X天）
- 整体最短修复时间: X 小时（XX组件）
- 整体最长修复时间: X 小时（XX组件，约X天）

## 重点发现
1. XX组件平均修复时间最长（X小时），主要原因：XXX
2. XX组件工单数量最多（X个），占比X%
3. XX组件平均修复最快（X小时）
```
